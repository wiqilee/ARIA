## Transport

| Property | Value |
|---|---|
| Protocol | MCP over HTTP (JSON-RPC 2.0) |
| Endpoint | `POST /mcp` |
| Content-Type | `application/json` |
| Discovery | `GET /` returns the MCP server info and the full tool registry |

A typical tool call looks like this:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "check_interactions",
    "arguments": { "...": "..." }
  }
}
```

The response always follows the JSON-RPC 2.0 envelope shape, with the tool's structured output in `result.content`.

## Tools

### `check_interactions`

Detect pairwise and N-drug interactions from a medication list.

**Input:**

```json
{
  "drugs": [
    { "name": "warfarin", "rxcui": "11289" },
    { "name": "aspirin",  "rxcui": "1191" }
  ],
  "patient_context": {
    "age": 72,
    "sex": "female",
    "ckd_stage": 3
  }
}
```

**Output:** `InteractionReport` containing the list of detected interactions, each with severity, type (pharmacokinetic or pharmacodynamic), and the affected drugs. Interactions that involve more than two drugs are surfaced with the full participant set, not just pairwise pieces.

### `explain_mechanism`

Provide mechanistic reasoning for a specific drug interaction at the molecular level: CYP enzyme competition, renal clearance, protein binding displacement, gut microbiome interference, and so on.

**Input:**

```json
{
  "drug_a": { "name": "warfarin", "rxcui": "11289" },
  "drug_b": { "name": "aspirin",  "rxcui": "1191" }
}
```

**Output:** `MechanisticExplanation` with the pathway in plain prose, the enzymes or transporters involved, and a clinical significance assessment generated by Gemini 2.5 Pro.

### `score_risk`

Calculate a personalized risk score on a 0 to 10 scale, adjusted for patient phenotype.

**Input:**

```json
{
  "interaction": {
    "drug_a": "warfarin",
    "drug_b": "aspirin",
    "type":   "pharmacodynamic"
  },
  "phenotype": {
    "age": 72,
    "sex": "female",
    "weight_kg": 65,
    "ckd_stage": 3,
    "hepatic_impairment": false,
    "smoking": false
  }
}
```

**Output:** `RiskScore` containing the numeric score, the risk-multiplier factors that were applied (renal, hepatic, age, weight), and a one-paragraph reasoning block explaining how the score was reached.

### `suggest_alternatives`

Suggest evidence-based drug substitutions to reduce interaction risk.

**Input:**

```json
{
  "drug":   { "name": "aspirin", "rxcui": "1191" },
  "reason": "triple anticoagulant risk with warfarin and fish oil",
  "patient_context": { "age": 72, "ckd_stage": 3 }
}
```

**Output:** `Alternatives` with a ranked list of substitution options, each annotated with trade-off analysis (efficacy, safety profile, contraindications) and PubMed citations.

### `build_interaction_graph`

Construct an N-drug interaction graph with hub drug identification and emergent multi-drug interaction detection.

**Input:**

```json
{
  "drugs": [
    { "name": "warfarin",   "rxcui": "11289" },
    { "name": "aspirin",    "rxcui": "1191"  },
    { "name": "omeprazole", "rxcui": "7646"  },
    { "name": "fish oil" }
  ]
}
```

**Output:** `InteractionGraph` with:

- **Nodes:** one per drug, with degree and centrality metrics
- **Edges:** one per pairwise interaction, weighted by severity
- **Hub drugs:** the drug or drugs responsible for the largest share of interactions in the graph
- **Cluster analysis:** groups of drugs that interact among themselves
- **Emergent interactions:** three-drug or larger interactions that pairwise logic cannot see (for example, the aspirin and warfarin and fish oil triple anticoagulant case)

### `compute_burden_scores`

Calculate cumulative clinical burden scores across all medications.

**Input:**

```json
{
  "drugs": [
    { "name": "diphenhydramine" },
    { "name": "oxybutynin" },
    { "name": "amitriptyline" }
  ]
}
```

**Output:** `BurdenScores` with:

- **Anticholinergic burden** on the Anticholinergic Cognitive Burden (ACB) scale, 0 to 3 per drug
- **Sedation load** on a 0 to 3 per-drug scale
- **QT prolongation risk** based on CredibleMeds classification
- **Per-drug contributions** so a clinician can see which medication is driving each cumulative score

### `model_temporal_cascade`

Model the timeline of risk emergence for drug interactions.

**Input:**

```json
{
  "drugs": [
    { "name": "warfarin" },
    { "name": "fluconazole" }
  ],
  "timeline": {
    "start_date":    "2026-01-01",
    "duration_days": 14
  }
}
```

**Output:** `CascadeModel` with daily risk projections, the day when peak risk is expected, and the intervention windows where dose adjustment or therapeutic drug monitoring would have the most impact.

### `generate_deprescribing_plan`

Produce a prioritized deprescribing plan with expected risk reduction at each step.

**Input:** `FullAnalysis`, the combined output of the previous tools.

**Output:** `DeprescribingPlan`, an ordered list of steps. Each step contains:

| Field | Description |
|---|---|
| `target_drug` | The drug to address at this step |
| `action` | `discontinue`, `taper`, `substitute`, or `dose_reduce` |
| `substitution` | Recommended replacement drug, when applicable |
| `monitoring` | Labs and clinical signs to track during the change |
| `expected_risk_delta` | Predicted change in the overall risk score after this step |

### `generate_report`

Assemble all analysis results into a single structured clinical report.

**Input:** `FullAnalysis`, the combined output of the previous tools.

**Output:** `ClinicalReport`, a formatted report with all sections (interactions, mechanisms, burden scores, temporal cascade, deprescribing plan), suitable for clinical review and for export to HTML or PDF by the frontend.

### `fhir_patient_medications`

Read active medications directly from any FHIR R4 endpoint via the `MedicationRequest` resource. This is the entry point for the SHARP-aware Path B integration described in [`docs/sharp-integration.md`](sharp-integration.md).

**Input:**

```json
{
  "patient_id":        "example-patient-1",
  "fhir_bearer_token": "eyJhbGciOiJSUzI1NiIs...",
  "fhir_server_url":   "https://hapi.fhir.org/baseR4"
}
```

All three fields are optional at the tool layer. The agent layer populates them from the A2A FHIR Context extension or the SHARP HTTP headers, in that order. When all three are empty, the tool falls back to the `FHIR_BASE_URL` and `FHIR_DEFAULT_PATIENT_ID` environment variables, then to the public HAPI FHIR sandbox without any `Authorization` header.

**Output:** `FhirMedicationList` with the resolved patient ID, the resolved FHIR base URL, the total count, and a list of medications. Each medication contains the RxNorm code, the display name, the medication status (active, on-hold, completed, etc.), and the dosage instruction text.

```json
{
  "patient_id":    "example-patient-1",
  "fhir_base_url": "https://hapi.fhir.org/baseR4",
  "total":         4,
  "medications": [
    {
      "rxnorm_code":  "855332",
      "display_name": "Warfarin Sodium 5 MG Oral Tablet",
      "status":       "active",
      "dosage_text":  "Take 5 mg by mouth once daily"
    }
  ]
}
```

See [`docs/sharp-integration.md`](sharp-integration.md) for the full FHIR context propagation rules, fallback ladders, and security invariants.

## Health Check

```
GET /health
```

Returns `200 OK` with a JSON body:

```json
{
  "service": "aria-mcp-server",
  "status":  "healthy",
  "version": "0.1.0"
}
```

The agent's matching health check is at `GET /health` on port 8000 and additionally reports MCP server connectivity:

```json
{
  "status":      "healthy",
  "service":     "aria-agent",
  "mcp_server":  "connected",
  "vertex_ai":   "reachable"
}
```

## Errors

All tools return standard JSON-RPC 2.0 error envelopes. The most common errors:

| Code | Meaning | Typical cause |
|---|---|---|
| `-32600` | Invalid request | Malformed JSON-RPC envelope |
| `-32601` | Method not found | Tool name is misspelled or not registered |
| `-32602` | Invalid params | Required field missing or wrong type |
| `-32603` | Internal error | Upstream API failure (OpenFDA, RxNorm, Vertex AI). Retry with backoff. |
| `-32000` | Server error (FHIR) | Underlying FHIR server returned a non-2xx response. The error data contains the upstream status code and a sanitized error message with no patient identifiers. |
